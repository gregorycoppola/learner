
# learner

An empirical companion to the paper:

> **Agent Completeness via Circuit Simulation:
> A Natural Proof that Transformer Agents are Universal Computers**
> Greg Coppola, PhD (March 2026)

The paper proves theoretically that a 4-layer transformer operating in a
read-compute-write agent loop is Turing complete. This project asks the
empirical follow-up question:

> Can gradient descent actually find those weights?

## Result

Yes. A 2-layer, 2-head transformer with d_model=32 learns to perfectly
simulate the binary incrementer Turing machine — 100% accuracy on all
transitions including carry — in 4 epochs of supervised training on 8056
examples.

## The Experiment

We use the binary incrementer TM: three states (scan_right, carry, done),
alphabet {0, 1, _}, MSB-first binary tape. The task is single-step
prediction: given a snapshot (tape, head position, state), predict the
next snapshot after one TM transition.

Training data is generated by running the TM forward on random integers
0-255, collecting every consecutive (before, after) step pair. The dataset
is balanced 50/50 carry/scan_right to prevent the model from ignoring the
minority carry state.

If the transformer learns the transition function, we can unroll the agent
loop and watch it run the full computation. This directly tests the open
question in Section 8.3 of the paper: whether trained transformers learn
weight configurations that approximate the circuit simulation structure —
not just that such weights exist, but that SGD finds them.

## Architecture

**Model:** TMTransformerCategorical — a standard transformer encoder with
three separate output heads:

- `value_logits (B, n, 3)` — tape symbol at each cell: {0, 1, _}
- `head_logits (B, n)` — head position (softmax over tape length)
- `state_logits (B, S)` — next state (softmax over state set)

**Encoding:** Each tape cell becomes one token of dimension `d = 3 + b + 1 + S`
where `b = ceil(log2(n+1))` and `S` = number of states:

    [val_oh(3) | pos(b) | is_head(1) | state_oh(S)]

The value slot uses a 3-class one-hot over {0, 1, _}. This is critical:
encoding blank as a separate class (not conflated with zero) is required
for correct supervision of transitions that write to blank cells.

**Hyperparameters:** 2 layers, 2 heads, d_model=32, lr=1e-3, batch_size=32,
Adam optimizer, 10000 samples, balanced carry/scan_right.

## Training

Loss is weighted cross-entropy applied separately to carry and scan_right
examples:

- scan_right examples: standard CE on all three heads
- carry examples: same CE multiplied by carry_weight (default 10x)

Both terms are fully differentiable. The carry weight ensures carry errors
receive proportionally stronger gradient signal despite carry being a
minority class in the natural distribution (~10%).

## Key Finding: The Encoding Bug

Six training approaches failed to move carry accuracy off 0.0% before the
root cause was identified. The bug was in the value encoding: using a
single bit (`1.0` for "1", `0.0` for "0" or "_"`) made blank cells
indistinguishable from zero cells. The carry transition `("carry", "_") →
write "1"` — the new-MSB case — was being given the wrong supervision
target. Once blank was given its own encoding class, the model solved the
full task in 4 epochs.

Failed approaches (all with the broken encoding):

    MSE regression, natural distribution       carry: 0.0%
    MSE regression, balanced sampling          carry: 0.0%
    MSE + curriculum / hard example mining     carry: 0.0%
    Categorical model + cross-entropy SFT      carry: 0.0%
    GRPO cold start                            carry: 0.0%
    SFT then GRPO (K=64)                       carry: 0.0%
    Weighted CE carry loss (non-differentiable) carry: 0.0%

After fixing the encoding:

    Weighted CE, 3-class encoding, epoch 4     carry: 100.0%

## Project Structure

    src/learner/core/
      machines.py        — TM definitions (binary incrementer)
      tm.py              — TM simulator
      data.py            — training data generation from TM traces
      encoding.py        — snapshot encoding (3-class value one-hot)
      model.py           — TMTransformerCategorical
      trainer_hybrid.py  — weighted CE training loop
      analysis.py        — per-feature accuracy breakdown

    src/learner/cli/     — CLI commands (hybrid, grpo, sft-grpo, analyze)
    src/learner/server/  — FastAPI server with SSE streaming
    reports/             — experiment logs

## Running

Start the server:

    learner server start

Train:

    learner hybrid run --samples 10000 --epochs 100 --carry-weight 10 --analyze-every 5

Analyze a checkpoint:

    learner analyze run --samples 1000
